## Bối cảnh

ZEN 1.0.15 đã unify pair schema thành `{ curve, pub, priv, address }`. Không còn `epub`/`epriv` riêng — `pub` dùng trực tiếp cho cả signing (ECDSA) lẫn ECDH (`zen.secret`). `pair.address` chứa EVM checksummed address ngay trong pair, không cần ethers.js để derive address nữa.

ZEN còn bổ sung `zen.recover(signedString)` — recover pub key từ compact signed string mà không cần biết trước pub. Đây là tư duy khác hẳn GUN cũ: GUN không có recover nên phải gửi pub kèm message. ZEN thì pub được suy ra từ chữ ký.

Akao vẫn dùng ZEN 1.0.11 (`package.json`), chưa bump lên 1.0.15.

## 3. Wave auth — chuyển sang tư duy ZEN (`zen.recover()`)

Thay đổi kiến trúc quan trọng nhất ngày hôm nay.

### Vấn đề với approach cũ (tư duy GUN)

GUN không có cơ chế recover pub từ chữ ký, nên phải gửi pub kèm message:
```js
// Cũ — gửi pub tường minh
send({ "~": session.pub, ":": ">" })      // request
send({ "~": pair.pub, "enc": enc })       // grant
send({ ":": "!>" })                       // deny
```

Receiver phải tin vào `parsed["~"]` — không có cách verify pub đó thực sự là người ký message.

### Approach mới (tư duy ZEN)

ZEN có `zen.recover(signedString)` → recover pub từ ECDSA recovery bit. Không cần gửi pub, receiver tự recover:

```js
// Mới — chỉ gửi chữ ký
const signed_req = await zen.sign(">", session)    // B gửi
// A nhận:
const pub_B = await zen.recover(signed_req)        // lấy session pub
const content = await zen.verify(signed_req, pub_B) // === ">"
```

**Grant flow:**
```js
// A gửi:
const secret = await zen.secret(pub_B, pair_A)
const enc    = await zen.encrypt(seed, secret)
const reply  = await zen.sign(enc, pair_A)    // ký ciphertext

// B nhận:
const pub_A  = await zen.recover(reply)
const enc    = await zen.verify(reply, pub_A)
const secret = await zen.secret(pub_A, session)
const seed   = await zen.decrypt(enc, secret)
```

**Deny:**
```js
const msg = await zen.sign("!>", pair_A)  // fallback: "!>" nếu không có pair
```

**Thay đổi trong code:**

| File | Thay đổi |
|------|---------|
| `authenticate/logic.js` | `request()` gửi `zen.sign(">", session)`; `handle()` dùng `recover()+verify()` |
| `authorize/logic.js` | `isrequest()` detect signed string format (`/^[0-9A-Za-z]{86}[01][:/]/`); `encode()` recover sessionPub từ chữ ký; thêm `deny()` ký `"!>"` |
| `authenticate/index.js` | `onwave()` dùng `parsed ?? message` fallback — signed string không phải JSON nên `parsed === null` |
| `authorize/index.js` | `wave()` async recover pub; `grant()` pass full signed request; `deny()` gọi `logic.deny()` |

**Security gain**: trước đây không verify được pub trong `"~"` có phải là người ký không. Giờ pub được cryptographically derived từ chữ ký — không thể giả mạo.

## 9. Tổng kết

Hai tư duy cần nhớ sau ngày hôm nay:

**Về whitepaper**: trong giao thức akao, maker/taker chỉ cần `pub` — không cần `epub` riêng vì ZEN đã unify. `pub` dùng cho cả signing (ECDSA) lẫn key agreement (ECDH). Mọi object `{ pub, epub }` trong whitepaper đều là di vật từ thời GUN.

**Về wave auth**: không cần gửi pub trong message. `zen.recover()` derive pub từ chữ ký. Đây là cách ZEN được thiết kế — trust là thứ crypto-verifiable, không phải thứ self-reported. Bất kỳ nơi nào trong akao còn pattern `{ "~": pub, ... }` để "khai báo" pub đều nên xem xét thay bằng signed messages + recover.
