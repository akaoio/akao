# Nợ kỹ thuật: Lock / Order / Trade — Lỗi mật mã và thiết kế

**Phát hiện**: 2026-04-08

---

## CRITICAL — `epub` vs `pub` trong `sea.secret()`

**File**: `src/core/Lock.js` (và mọi caller tương lai)

GunDB SEA dùng hai loại public key hoàn toàn khác nhau:

- `pair.pub` → **Ed25519** — chỉ dùng cho `sea.sign()` / `sea.verify()`
- `pair.epub` → **Curve25519** — chỉ dùng cho `sea.secret()` / `sea.encrypt()`

`sea.secret()` **bắt buộc phải nhận `epub`**, không phải `pub`. Nếu truyền `pub` vào `sea.secret()`, kết quả là sai hoàn toàn (silent wrong output) hoặc crash.

**Đã sửa**: White paper cập nhật toàn bộ 17 chỗ từ `sea.secret(P.pub, ...)` → `sea.secret(P.epub, ...)`.

**Cần làm**: Mọi implementation của `Lock`, `Order`, `Trade` phải dùng `epub` mọi nơi gọi `sea.secret()`.

## CRITICAL — `get async method()` là syntax JavaScript không hợp lệ

**File**: `src/core/Lock.js` (bản draft cũ)

```js
// SAI — JS không hỗ trợ async getter
get async secret() { ... }
get async address() { ... }
```

Parser đọc `async` là tên property, sau đó báo lỗi `'(' expected`. Phải dùng async method bình thường:

```js
async secret() { ... }
async address() { ... }
```

## CRITICAL — Constructor trả về object thay vì throw

**File**: `src/core/Lock.js` (bản draft cũ)

```js
constructor(...) {
    if (!payer || !escrow || !recipient) return { error: "invalidInput" }
}
```

Trong JS, return một non-primitive từ constructor sẽ thay thế `this` bằng object đó. `new Lock({})` trả về `{ error: "invalidInput" }`, không phải instance Lock. Khi caller gọi `lock.address()` → crash silently.

```js
// ĐÚNG — throw rõ ràng
if (!pair || !epub || !xpub || !trade) throw new Error("Lock: missing required inputs")
```

## HIGH — TL và CL dùng cùng index — rủi ro correlation on-chain

**Gốc rễ**: `sha256(secret + ":" + tradeId)` dùng cho cả TL và CL → `index_TL === index_CL`.

Bất kỳ ai biết xpub của recipient và affiliate đều có thể xác nhận hai địa chỉ escrow thuộc cùng một trade (vì cùng index, khác tree).

**Fix** — domain separator khác nhau cho từng lock:

```js
index_TL = parseInt(sha256(secret + ":TL:" + tradeId).slice(0,8), 16) & 0x7fffffff
index_CL = parseInt(sha256(secret + ":CL:" + tradeId).slice(0,8), 16) & 0x7fffffff
```

**Đã sửa**: White paper cập nhật toàn bộ — Step 3, Step 5, Platform refund, Section 4.3.

## HIGH — xpub của recipient/affiliate phải lấy từ `gun.user(pub)`

Fetch xpub từ path tùy ý là lỗ hổng injection:

```js
// NGUY HIỂM
gun.get("someArbitraryPath").get("xpub").once(...)
```

Nếu xpub bị thay thế bởi attacker, payer deposit vào address thuộc tree của attacker — recipient không bao giờ có thể spend (không có xprv). Funds mất vĩnh viễn.

```js
// AN TOÀN — data trong gun.user(pub) được SEA ký với private key của owner
// Gun từ chối mọi write không khớp signature
gun.user(recipient.pub).get("xpub").once(...)
gun.user(affiliate.pub).get("xpub").once(...)
```

**Cần làm**: Khi implement `Trade` và `Lock`, mọi xpub fetch phải qua `gun.user(pub)`. Cần thiết kế Pen soul cho trade record ký bởi maker, không phải path tùy ý.

## HIGH — `tradeId` chứa `Date.now()` — phải store, không recompute

```js
tradeId = sha256(orderId + M.pub + T.pub + timestamp)  // timestamp = Date.now() lúc match
```

Cả payer và recipient đều cần đến **đúng một** `tradeId` để tính index. Nếu mỗi bên recompute với timestamp khác nhau → index khác nhau → recipient không bao giờ derive được spending key đúng → funds bị khoá vĩnh viễn.

`tradeId` phải được compute một lần duy nhất lúc match, store vào Gun trade record, và cả hai bên đọc từ Gun.

## MEDIUM — Principle of least privilege trong constructor

`Lock` hiện nhận `payer` (full Access object) và `platform` (full config object). Lock chỉ cần:
- `pair` — SEA pair của payer (cho `sea.secret`)
- `epub` — Curve25519 public key của platform
- `xpub` — BIP-32 extended public key của recipient
- `trade` — tradeId string

```js
// Đúng hơn — truyền đúng thứ cần, không thừa
new Lock({ pair, epub, xpub, trade, type: "TL" })
```

Tránh expose toàn bộ object vào một primitive cryptographic class.

## MEDIUM — `sha256()` custom chỉ đúng với ASCII input

`src/core/Utils/crypto.js` dùng `str.charCodeAt(i)` → chỉ chuẩn với ASCII.

Inputs của Lock đều là ASCII (base64 SEA secrets, hex tradeId, literal separators) nên hiện tại an toàn. Nhưng nếu ai truyền UTF-8 string vào → sha256 sai, silent.

**Cần làm**: Document rõ trong Lock source code rằng tất cả inputs phải là ASCII-safe strings.

## LOW — Đụng độ naming: `pair.pub` (signing) vs `Wallet.public` (EVM address)

Hai khái niệm hoàn toàn khác nhau dùng tên gần nhau:
- `Access.pair.pub` → SEA Ed25519 signing key (Gun identity)
- `Wallet.public` (trong `src/core/Wallet.js`) → EVM public key (address)

Khi đọc code Lock/Trade, dễ nhầm. Cần giữ consistent naming trong implementation.

## HIGH — Order soul Pen: `sign: true` không chặn overwrite từ user bất kỳ ✅ RESOLVED

**Giải pháp**: Thêm `pub8 = M.pub.slice(0, 8)` vào segment 3 của key. Pen validate `PRE(R[5], seg3)` — writer's pub phải bắt đầu bằng pub8 trong key. Key format mới: `<candle>:<item_slug>:<type>:<pub8>:<nonce>`. Xem Section 6.1 và 6.2 white paper.

## HIGH — FP → TL/CL KHÔNG phải atomic transaction

White paper gọi là "atomic transition" — đây là **sai**. Trên EVM:
- TL deposit là tx1
- CL deposit là tx2
- Platform fee là tx3

Ba tx độc lập. Nếu tx2 fail sau khi tx1 thành công → TL có tiền nhưng CL trống → trade ở partial state. **Implementation phải handle từng receipt và có recovery path**.

**Đã sửa**: White paper đã xóa từ "atomic", thay bằng cảnh báo rõ.

## HIGH — FP race condition (front-running)

Maker có full control của FP wallet (by design). Sau khi Taker accept order (match):
1. Taker đã check `balance >= requiredAmount` ✓
2. Maker có thể drain FP wallet trong cùng block
3. Payer cố deposit từ FP → fail

**Đây là giới hạn của thiết kế không dùng smart contract.**

**Mitigation**: Taker nên re-verify FP balance tại thời điểm match (không chỉ lúc discover), và có timeout để cancel trade nếu deposit không xảy ra trong 10 phút.

---

## HIGH — Recipient có thể write xpub sai/crafted vào `gun.user(pub)`

Gun đảm bảo data trong `gun.user(pub)` được SEA-ký bởi chủ sở hữu. Nhưng chủ sở hữu có thể write **bất kỳ xpub nào** vào đó — kể cả xpub của một HD tree mà không ai có xprv.

Nếu recipient write xpub crafted:
1. Payer deposit vào TL derived từ xpub đó
2. Recipient không có xprv → không withdraw được
3. Platform không biết secret của tree đó → không arbitrate được
4. **Funds bị khoá vĩnh viễn**

**Mitigation**: Platform phải validate `xpub_MP` match derivation expected trước khi trade vào deposit phase:
```js
const expected_root = HDNodeWallet.fromSeed(sha256(await sea.secret(P.epub, recipient.pair)))
const expected_xpub = expected_root.neuter().extendedKey
assert(published_xpub === expected_xpub, "xpub mismatch — possible malicious recipient")
```

## MEDIUM — FP index thiếu domain separator

```js
// THIẾU — không có domain prefix, inconsistent với TL/CL
index_order = parseInt(sha256(orderId).slice(0,8), 16) & 0x7fffffff

// ĐÚNG — "FP:" domain separator giống phong cách TL/CL
index_order = parseInt(sha256("FP:" + orderId).slice(0,8), 16) & 0x7fffffff
```

FP và TL (khi Maker là recipient của sell order) dùng cùng root HD. Dù xác suất collision gần như 0 trong thực tế (entropy nguồn khác nhau), domain separator là bắt buộc về nguyên tắc.

**Đã sửa**: White paper cập nhật.

## MEDIUM — BIP-32 non-hardened child key leakage

BIP-32 non-hardened derivation có property: `child_private_key + parent_xpub → parent_private_key`.

Nếu spending key của TL hoặc CL bị leak (console.log, error logging, memory dump):
- Attacker có `child_priv` (leaked) + `xpub_recipient` (public) → derive `parent_priv` → compromise **toàn bộ** HD tree của recipient trong context escrow.

**Rule**: Spending keys (`TL.privateKey`, `CL.privateKey`) phải:
1. Chỉ tồn tại trong scope ngắn nhất có thể
2. Xóa khỏi memory ngay sau `sendTransaction()`
3. Không bao giờ log, store, hay pass vào function khác
