# Technical Debt: DB.js SQL Sync Implementation

**Discovered**: 2026-04-09  
**PR**: #49 (fix/data-sync-to-sql)  
**Impact**: High — affects offline-first UX, data consistency, and long-term maintainability

---

## 1. Stale Data Accumulation

### Issue
No cleanup mechanism for removed items. If an item is deleted from the catalog or renamed, old entries persist indefinitely in SQL/IDB.

### Current Behavior
- `transform()` only validates current paths — never removes stale rows
- `$rebuildFromIDB()` uses `INSERT OR IGNORE` — preserves old data
- `$syncCatalog()` only adds/updates — never deletes

### Risk
Over time, SQL database grows with obsolete items. Queries may return deleted products, breaking UX.

### Solution Options
1. **Timestamp-based cleanup**: Track `last_seen_at` per item, delete rows older than N days
2. **Full reconciliation**: Periodic compare SQL vs catalog manifest, delete orphans
3. **Versioned catalog**: Embed version in `meta.json`, rebuild SQL on major version bump

**Recommendation**: Option 1 (timestamp) — lowest overhead, handles most cases.

---

## 2. Schema Migration Strategy Missing

### Issue
No versioning or migration system. If table schema changes (e.g., add `stock` column to `catalog_items`), existing SQL data becomes incompatible.

### Current Behavior
- `SCHEMA_*` constants in `DB/schemas/` are unversioned
- `DB.sql()` checks `_schemas` Set but not schema version
- Schema mismatch = silent failures or incorrect queries

### Risk
Future schema changes require manual OPFS deletion by users. No automated migration path.

### Solution
1. Add `DB_VERSION` constant to `DB.js`
2. Store version in SQL: `CREATE TABLE IF NOT EXISTS _meta (key TEXT PRIMARY KEY, value TEXT)`
3. On version mismatch: run migration SQL or clear + rebuild

**Recommendation**: Implement before first production release (currently in development phase).

## 3. Memory Leak Risk in `DB._pending`

### Issue
`DB._pending` array grows unbounded if `_flush()` fails repeatedly (e.g., SQL worker crash before microtask fires).

### Current Behavior
- `$syncToSQL()` pushes to `_pending` without size check
- Watchdog in `SQL/call.js` clears queue on timeout, but `_pending` persists
- Re-accumulation possible if `_flush()` never completes

### Risk
Prolonged worker failure → `_pending` grows to thousands of items → OOM in browser.

### Solution
Add max size limit with warning:

```javascript
static $syncToSQL(path, data) {
    const op = transform(path, data)
    if (!op) return
    
    if (DB._pending.length >= 10000) {
        console.warn("[DB.$syncToSQL] queue overflow, dropping oldest entries")
        DB._pending.splice(0, 5000) // drop half
    }
    
    DB._pending.push(op)
    // ... rest of logic
}
```

**Recommendation**: Add immediately — safety guardrail with minimal cost.

## 4. `$syncCatalog` Startup Performance Hit

### Issue
On first load, `$syncCatalog()` fetches **every item in every locale** (19 locales × ~hundreds of items = thousands of requests).

### Current Behavior
- Runs after `$rebuildFromIDB()` completes
- No incremental sync — fetches all items every cold start if SQL is empty
- Uses `DB._yield()` to avoid blocking, but still heavy network load

### Risk
Slow first load on poor connections. Lighthouse "Time to Interactive" penalty.

### Solution Options
1. **Skip if meta hash unchanged**: Compare `meta.json` hash between runs, skip sync if unchanged
2. **Priority queue**: Sync visible items first (homepage), defer rest
3. **Service Worker prefetch**: Move sync to background after initial paint

**Recommendation**: Option 1 (hash check) — simplest, covers common case (catalog unchanged).

## 5. Race Condition: Duplicate Inserts During Rebuild

### Issue
If `DB.get()` fires during `$rebuildFromIDB()`, the same item may be inserted twice:
1. Rebuild chunk reads from IDB → calls `_syncInsert()`
2. User navigates → route calls `DB.get()` for same item → calls `$syncToSQL()`

### Current Behavior
- `INSERT OR IGNORE` prevents SQL errors, but both paths do work
- No coordination between rebuild and user-triggered fetches

### Risk
Wasted CPU/IO during startup. Not critical (idempotent), but inefficient.

### Solution
Add flag to skip sync during rebuild:

```javascript
static _rebuilding = false

static async $rebuildFromIDB() {
    DB._rebuilding = true
    // ... existing logic
    DB._rebuilding = false
}

static $syncToSQL(path, data) {
    if (DB._rebuilding) return // skip, rebuild will handle it
    // ... rest of logic
}
```

**Recommendation**: Low priority — optimize after measuring real-world impact.

## 6. No Exposed SQL Reset Method

### Issue
Developers have no way to clear SQL database for testing/debugging without manually deleting OPFS files.

### Current Behavior
- `DB.sql()` initializes DB but never clears it
- Previous `DROP TABLE` logic was removed in this PR
- Only option: browser DevTools → Storage → OPFS → manual deletion

### Risk
Frustrating DX during development. Stale test data persists across refreshes.

### Solution
Add dev-only method:

```javascript
static async clearSQL() {
    if (!DEV) throw new Error("clearSQL is dev-only")
    const db = await DB.sql()
    await db.exec("DROP TABLE IF EXISTS game_items; DROP TABLE IF EXISTS catalog_items;")
    DB._schemas.clear()
    console.log("[DB.clearSQL] tables dropped, schemas cleared")
}
```

**Recommendation**: Add immediately — unblocks local development.

## 7. Hardcoded Timeouts in `SQL/call.js`

### Issue
Timeout values are hardcoded constants. No configuration for slow devices or CI environments.

### Current Behavior
```javascript
const TIMEOUTS = {
    open: 30000,
    batch: 30000,
    // ...
}
```

### Risk
- Too short → false positives on slow hardware (e.g., old Android)
- Too long → hangs feel unresponsive on modern devices

### Solution
Make configurable via `SQL` constructor options:

```javascript
class SQL {
    constructor({ name, timeouts = {} }) {
        this.timeouts = { ...DEFAULT_TIMEOUTS, ...timeouts }
    }
}
```

**Recommendation**: Medium priority — wait for real-world timeout reports before adding complexity.

## 8. Infinite Recursion Risk in `pump()`

### Issue
`pump()` calls itself recursively via callback. If queue is huge (10k+ items), could stack overflow.

### Current Behavior
```javascript
threads.queue({
    callback: (response, error) => {
        // ...
        pump() // recursive call
    }
})
```

### Risk
Unlikely in practice (microtasks drain queue incrementally), but theoretically unbounded recursion.

### Solution
Replace recursion with while loop:

```javascript
async function pump() {
    while (queue.length > 0 && !processing) {
        processing = true
        const { method, params, resolve, reject } = queue.shift()
        // ... existing logic
        await new Promise(r => threads.queue({ callback: r }))
        processing = false
    }
}
```

**Recommendation**: Low priority — edge case, refactor if profiling shows stack issues.

## 9. No Progress Visibility for `$syncCatalog`

### Issue
Users have no feedback during catalog sync. App appears frozen on slow connections.

### Current Behavior
- `$syncCatalog()` runs silently in background
- No events, no progress bar, no console logs (unless error)

### Risk
Poor UX on first load. Users may refresh, aborting sync and forcing restart.

### Solution
Emit progress events:

```javascript
static async $syncCatalog() {
    // ...
    let synced = 0
    const total = meta.children * locales.length
    
    for (const key of keys) {
        await DB.get(path)
        synced++
        if (synced % 50 === 0) {
            Context.set({ catalogSyncProgress: synced / total })
        }
    }
}
```

**Recommendation**: High priority for production — critical for UX.

## 10. `_yield()` Has No Timeout

### Issue
If `scheduler.yield()`, `MessageChannel`, and `setTimeout` all fail to resolve, `_yield()` hangs forever.

### Current Behavior
```javascript
static _yield() {
    if (typeof scheduler !== "undefined" && scheduler.yield) return scheduler.yield()
    // ... fallbacks
    return new Promise((r) => setTimeout(r, 0))
}
```

### Risk
Extremely unlikely (requires broken browser APIs), but would deadlock rebuild/sync.

### Solution
Add timeout wrapper:

```javascript
static _yield() {
    const promise = /* ... existing logic */
    return Promise.race([
        promise,
        new Promise((_, reject) => setTimeout(() => reject(new Error("yield timeout")), 5000))
    ])
}
```

**Recommendation**: Low priority — defensive coding, unlikely to trigger.
