## 1. `ui-user` avatar flicker on page reload

**Symptom:** After a hard reload, the `ui-user` button in the nav takes a noticeable moment before
it switches from the default icon to the user's saved identicon avatar. The delay is visible as a
flash of the generic icon before the correct avatar renders.

**Root cause (hypothesis):** `ui-user` renders synchronously on mount, but resolving the avatar
seed requires async work — `Access.get("seed")` must be available, then two rounds of
`sea.work()` hashing must complete before the seed is passed to `ui-identicon`. If the auth
session is not yet rehydrated when the component connects, the identicon is skipped entirely and
only re-rendered once `Access.on("avatar")` fires, which is too late to avoid the flash.

**What to investigate:**

- How early `Access` rehydrates its session state on boot (check `Launcher.js` / thread init order)
- Whether `sea.work()` can be precomputed or cached before the component mounts
- Whether a CSS skeleton / placeholder on `ui-user` would mask the flash cheaply while the real
  seed resolves

**Files to look at:**

- `src/UI/components/user/index.js` — `render()` and `connectedCallback`
- `src/UI/components/user/logic.js` — `Logic.seed()`
- `src/core/Access.js` — session rehydration timing
- `src/core/Launcher.js` — boot sequence

## 2. `ui-identicon` renders a different avatar when `data-size` changes

**Symptom:** Changing the `data-size` attribute on `ui-identicon` (e.g. from `5` to `7`) produces
a completely different-looking avatar for the same seed, instead of the expected behaviour of
showing the same identity at higher grid resolution (more detail, same core pattern).

**Root cause:** The bit-position formula uses `adjustedSize` as the row stride:

```js
const bitPosition = (row * adjustedSize + col) % 32
```

When `adjustedSize` changes (e.g. 5 → 7), every cell lands on a different bit of the same 32-bit
hash — so the entire pattern changes. Ideally the stride should be a fixed constant so that the
same `(row, srcCol)` pair always maps to the same bit regardless of the requested grid size.

**Constraint — even sizes:** The current code forces even sizes up to the next odd number
(`size % 2 === 0 ? size + 1 : size`) to guarantee a true centre column for mirroring. If even
sizes are ever supported natively, the axis of symmetry should be the left-hand centre column
(`ceil(size/2) - 1`), not an auto-bumped odd size.

**Proposed fix (to validate):** Replace the variable stride with a fixed constant (e.g. `4`),
decouple it from `adjustedSize`, and support even sizes natively via `leftCols = Math.ceil(size/2)`:

```js
const leftCols  = Math.ceil(size / 2)
const bitPos    = (row * 4 + srcCol) % 32   // stride = 4, fixed
const isFilled  = ((hash >> bitPos) & 1) === 1
```

This covers grids up to 7 rows × 4 srcCols = 28 unique bit positions — safely within 32 bits
with no wrapping collision. Needs visual verification across multiple seeds and sizes before
committing, since it will change the rendered appearance of all existing stored avatar IDs.

- `src/UI/components/identicon/index.js` — `generateIdenticon()`
