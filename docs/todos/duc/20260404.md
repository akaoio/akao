## Chương I — Xây Nền Móng: FS/OPFS Isomorphic

### Bối cảnh

Trước khi P2P có thể hoạt động, cần một thứ tưởng chừng đơn giản: **đọc và ghi file**.

Trên Node.js, ta có `fs`. Trên browser, ta có `OPFS` — Origin Private File System, một API mới của trình duyệt cho phép web app lưu trữ dữ liệu lớn như một filesystem thật sự. Vấn đề: hai môi trường, hai API hoàn toàn khác nhau.

Giải pháp: `FS.js` — một cánh cổng duy nhất. Gọi `FS.write()`, không cần biết đang ở đâu. Hệ thống tự biết nên dùng gì.

```
FS.js  ──► FS/write.js, FS/load.js, FS/dir.js, ...
               └── shared.js
                     ├── NODE    → await import("fs")
                     └── BROWSER → new OPFS()
```

Kiến trúc sạch. Triển khai xong. Nhưng ẩn trong đó có hai tên quái vật đang ngủ.

---

### ⚔️ Boss 1 — Kẻ Lật Mặt: `data?.abi || data`

**Cấp độ:** Thấp — nhưng nguy hiểm vì vô hình

**Phát hiện lúc:** Review session 2026-04-05

Trong `FS/load.js`, sau khi parse JSON xong, có một dòng code tưởng như vô hại:

```js
return data?.abi || data
```

Ý định ban đầu tốt đẹp: *"Nếu file JSON có field `abi`, chỉ trả về mảng ABI thôi — consumer cần ABI, không cần metadata xung quanh."* Đây là pattern phổ biến với smart contract ABI files trong Ethereum.

Nhưng JavaScript có một cái bẫy rất quen thuộc.

`[]` — mảng rỗng — là **truthy**.

Khi `data.abi = []` (contract mới deploy, chưa có function nào), biểu thức `data?.abi || data` trả về... `[]`. Không phải full object. Không phải error. Chỉ là một mảng rỗng im lặng, và caller nhận về dữ liệu cụt mà không hề hay biết.

*Đây là loại bug nguy hiểm nhất: không crash, không log, chỉ trả về sai.*

**Hạ gục:** Thay bằng kiểm tra explicit:

```js
// Trước — lật mặt khi abi là []
return data?.abi || data

// Sau — trung thực hoàn toàn
return data?.abi !== undefined ? data.abi : data
```

Giờ `abi: []` trả về đúng `[]`. `abi: null` trả về full object. Không còn góc khuất.

### ⚔️ Boss 2 — Kẻ Hủy Diệt Im Lặng: Binary Corruption

**Cấp độ:** 🔴 Critical — và chưa từng được document

**Phát hiện lúc:** Review session 2026-04-05, khi trace luồng dữ liệu từ `threads/torrent.js` ngược về `FS/load.js`

Đây là câu chuyện về một cái bẫy ẩn sâu nhất trong toàn bộ §1 đến §5.

`threads/torrent.js` có nhiệm vụ seed các file `.torrent` ra P2P swarm:

```js
const bytes = await FS.load(filePath)  // filePath = ["statics", "chains", "eth.torrent"]
torrent.seed(bytes, { announce: torrent.pool })
```

Nhìn qua, bình thường. Nhưng hãy theo dấu xuống tận `FS/load.js`, đến đoạn Node.js path:

```js
text = fs.readFileSync(_path, "utf8")
```

**"utf8".**

`.torrent` files không phải UTF-8. Chúng là **bencode** — định dạng binary của BitTorrent, nơi mà mỗi byte đều có ý nghĩa chính xác. Khi Node.js đọc binary data bằng `"utf8"`, các byte không hợp lệ bị replace bằng ký tự `U+FFFD` — dấu hỏi unicode. Dữ liệu bị biến dạng không thể khôi phục.

Browser path còn tệ hơn: `opfs.read()` trả về `ArrayBuffer`, rồi code làm:

```js
text = new TextDecoder().decode(buf)
```

Cũng là UTF-8 decode. Cũng corrupt.

Kết quả: `torrent.seed()` nhận một string rác. WebTorrent không crash — nó âm thầm thất bại. `.catch(() => {})` nuốt lỗi. **Toàn bộ §3b + §5 pipeline bị broken từ ngày đầu mà không ai biết.**

*Đây là Boss ẩn hình. Hệ thống chạy, không có error, nhưng không có gì được seed.*

**Hạ gục:** Thêm whitelist extension và binary detection:

```js
const textExts = ["json", "yaml", "yml", "csv", "tsv", "txt", "md", "html", "js", "css", "hash"]
const fileExt  = _path.match(/\.\w+$/)?.[0]?.slice(1).toLowerCase() || ""

// Node.js
if (textExts.includes(fileExt)) text = fs.readFileSync(_path, "utf8")
else return new Uint8Array(fs.readFileSync(_path))  // binary nguyên vẹn

// Browser — HTTP
if (isBinary) {
    const buf = await response.arrayBuffer()
    if (opfs) opfs.write(path, new Uint8Array(buf)).catch(() => {})
    return new Uint8Array(buf)
}
```

File `.torrent` giờ được đọc như binary. `Uint8Array` đến tay `torrent.seed()` nguyên vẹn. P2P pipeline thực sự hoạt động.

## Chương II — Bộ Nhớ Không Bao Giờ Mất: Torrent Persistence

### Bối cảnh

WebTorrent là kẻ hay quên. Đóng tab — mọi torrent đang chạy biến mất. Mở lại — phải add lại từ đầu.

§2 giải quyết điều này: mỗi khi add hoặc seed một torrent thành công, lưu state xuống filesystem. Mỗi khi startup, đọc lại và restore.

```
torrent/state/
├── abc123def456...json   ← state của torrent 1
├── 789xyz000aaa...json   ← state của torrent 2
└── ...
```

Mỗi file chứa đủ thông tin để resume:

```json
{
    "infoHash":  "abc123...",
    "magnetURI": "magnet:?xt=urn:btih:abc123...",
    "name":      "eth.json",
    "addedAt":   1712345678901
}
```

Kiến trúc đúng. Nhưng có một tên phá hoại ẩn trong implementation.

### ⚔️ Boss 3 — Kẻ Xóa Nhầm: State File Leak

**Cấp độ:** 🔴 High

Khi user remove một torrent, `remove.js` cần xóa state file tương ứng. Vấn đề: `remove(id)` nhận `id` có thể là *bất cứ thứ gì* — magnetURI, infoHash, hoặc key nội bộ của WebTorrent.

State files được đặt tên theo **infoHash** — chuỗi 40 ký tự hex. Nhưng code cũ làm thế này:

```js
// Trước — sai
$delete(id).catch(() => {})
```

Nếu `id = "magnet:?xt=urn:btih:abc123..."`, code cố xóa file có tên... `magnet:?xt=urn:btih:abc123....json`. File đó không tồn tại. `.catch(() => {})` nuốt lỗi. State file `abc123....json` còn đó mãi mãi, không bao giờ được dọn.

*Mỗi lần remove một torrent: leak một file. Mỗi restart: torrent đó lại được restore vào memory dù user đã xóa.*

**Hạ gục:** Resolve infoHash **trước** khi WebTorrent xóa torrent khỏi internal state:

```js
// Bước 1: Tìm infoHash thật sự từ bất kỳ dạng id nào
let infoHash = /^[0-9a-f]{40}$/i.test(id) ? id : null
if (!infoHash) {
    for (const [, torrent] of this.torrents) {
        if (torrent?.infoHash === id || torrent?.magnetURI === id) {
            infoHash = torrent.infoHash
            break
        }
    }
}

// Bước 2: WebTorrent remove (torrent vẫn còn trong this.torrents tại thời điểm này)
client.remove(id, options, (error) => {
    // Bước 3: Xóa đúng file
    if (infoHash) $delete(infoHash).catch(() => {})
})
```

Giờ bất kể bạn remove bằng cách nào — state file luôn được dọn sạch.

### ⚔️ Boss 4 — Kẻ Ngủ Đông: `restore()` Infinite Hang

**Cấp độ:** 🟡 Medium — nhưng có thể trở thành 🔴 khi integrate

`restore()` đọc danh sách torrents đã save và add lại vào WebTorrent. Logic đúng. Nhưng implementation có một cái bẫy chết người:

```js
// Trước — sequential await
for (const { magnetURI, infoHash } of saved) {
    const torrent = await this.add(magnetURI)  // ← đây
    results.push({ infoHash, ok: true })
}
```

`this.add(magnetURI)` resolve khi nào? Khi WebTorrent tìm được peer và nhận được metadata qua DHT network. Nếu không có peer nào đang hold torrent đó — Promise **không bao giờ resolve**.

Nếu torrent đầu tiên trong danh sách là một torrent cô đơn không có peer, `restore()` đứng yên ở đó mãi mãi. Torrent thứ hai, thứ ba — không bao giờ được add. Startup flow bị block hoàn toàn.

*Tưởng tượng: server khởi động lúc 3 giờ sáng sau một deploy, network chậm, không có peer nào online. App load mãi không xong.*

**Hạ gục:** Fire-and-forget. Không `await`, không block:

```js
// Sau — launch tất cả concurrently, không chờ ai
for (const { magnetURI, infoHash } of saved) {
    this.add(magnetURI).catch((error) =>
        console.error("Failed to restore torrent:", infoHash, error)
    )
}
```

Startup không còn phụ thuộc vào DHT network. Torrents restore dần dần trong background. Lỗi được log, không bị nuốt.

> **Ghi chú thám tử:** `restore()` hiện chưa được gọi ở đâu trong codebase — exported nhưng chưa invoke. Boss này đang ngủ đông. Nhưng khi §6 integrate, nó sẽ thức dậy — và lần này chúng ta đã sẵn sàng.

## Chương III — Dấu Vân Tay Của Dữ Liệu: Build Hash → BEP 3

### Bối cảnh

Để P2P sync hoạt động, mỗi file phải có một **fingerprint** duy nhất. Nếu fingerprint khớp — không cần download lại. Nếu khác — fetch version mới.

Trước đây, fingerprint là SHA-256 — 64 ký tự hex, thuần túy cryptographic. Tốt cho bảo mật, nhưng vô nghĩa với BitTorrent.

§3 thay bằng **BEP 3 v1 infohash** — chuẩn SHA-1 infohash của BitTorrent. 40 ký tự. Và giờ fingerprint của một file *chính là* địa chỉ để tìm file đó trên P2P swarm.

Một fingerprint, hai công dụng: detect change + P2P routing.

---

### ⚔️ Boss 5 — Kẻ Hỗn Loạn: Non-Deterministic Hash trên Linux

**Cấp độ:** 🟡 Medium — âm thầm phá vỡ reproducibility

Trên macOS, `fs.readdirSync()` trả về files theo thứ tự alphabetical. Trên Linux ext4, thứ tự phụ thuộc vào internal hash table của filesystem — **không deterministic giữa các máy**.

Hash của một multi-file directory torrent phụ thuộc vào thứ tự các files. Cùng một directory, cùng một nội dung — Linux CI trả về hash khác macOS dev machine.

*Build trên Mac: hash A. Deploy lên Linux server: hash B. User nhận thấy "có gì đó thay đổi" và fetch lại toàn bộ statics. Mỗi lần deploy.*

**Hạ gục:** Một dòng code:

```js
jsonFiles.sort()  // alphabetical, cross-platform
```

Trước khi build fileDescriptors — sort. Mọi môi trường, mọi OS, cùng một hash.

### ⚔️ Boss 6 — Người Gác Mù: `_.hash` Không Nhìn Thấy Subdirectories

**Cấp độ:** 🔴 Critical — phá vỡ toàn bộ change detection

Đây là Boss tinh vi nhất. Không phải bug trong một dòng code — mà là **design flaw trong cả một hệ thống**.

`update.js` — thread chạy mỗi giờ, kiểm tra xem có gì thay đổi trên server không — hoạt động như sau:

```js
const paths = [[], ["statics"], ["statics", "locales"], ["statics", "items"]]

for (const path of paths) {
    const hash = [...path, "_.hash"]
    const before = await Indexes.Hashes.get(hash).once()
    const after  = await DB.get(hash)

    if (before === after) {
        skippables.push(path)  // ← skip toàn bộ children nếu hash không đổi
        continue
    }
    // re-fetch...
}
```

Logic: nếu `statics/_.hash` không đổi → skip `statics/locales`, `statics/items`, mọi thứ bên trong. Hợp lý — tại sao kiểm tra chi tiết nếu tổng thể không thay đổi?

Nhưng với BEP 3, `statics/_.hash` được tính như thế này:

```js
if (fileDescriptors.length > 0) {
    // Chỉ hash các file trực tiếp trong statics/ (ví dụ: trackers.json)
    const { v1 } = await hash(fileDescriptors, "statics")
    await FS.write([...path, "_.hash"], v1)
    // ← KHÔNG bao gồm statics/chains/, statics/currencies/, ...
}
```

Bức tranh toàn cảnh:

```
statics/
├── trackers.json          ← _.hash chỉ cover cái này
├── chains/
│   ├── eth.json           ← THAY ĐỔI TRONG ĐÂY...
│   └── eth.hash
└── _.hash                 ← ...nhưng cái này KHÔNG ĐỔI
```

Khi `statics/chains/eth.json` được update trên server: `statics/chains/_.hash` thay đổi. Nhưng `statics/_.hash` — cái mà `update.js` đang canh gác — vẫn như cũ.

*`update.js` nhìn vào `statics/_.hash`, thấy "bình thường", thêm `["statics"]` vào skippables, và bỏ qua toàn bộ chains, currencies, locales. Data stale mãi mãi, không ai biết tại sao.*

**Hạ gục:** Tách biệt hai mục đích:
- **`_.torrent`** = BT infohash của direct files → dùng để **seed** (giữ nguyên)
- **`_.hash`** = combined hash của TẤT CẢ children → dùng để **detect change** (fix)

```js
if (fileDescriptors.length > 0) {
    // _.torrent: BT infohash của direct files — P2P seeding
    const { bytes: infoBytes } = await hash(fileDescriptors, dirName)
    await FS.write([...path, "_.torrent"], wrapTorrent(infoBytes))

    // _.hash: combined của direct files + subdir hashes — change detection
    // childHashes đã có cả hai: per-file hashes (line 69) + subdir _.hash values (line 90)
    const combined = encoder.encode(childHashes.sort().join(""))
    const { v1: dirHash } = await hash(combined, dirName)
    await FS.write([...path, "_.hash"], dirHash)
}
```

Giờ `statics/_.hash` phản ánh toàn bộ subtree. Thay đổi bất kỳ đâu trong `statics/**` → `statics/_.hash` đổi → `update.js` phát hiện → re-fetch đúng chỗ.

## Chương IV — Sinh Ra `.torrent`: Build Time Generation

### Bối cảnh

Để torrent thread có thể seed content, nó cần file `.torrent` — bản mô tả cấu trúc của torrent (tên file, kích thước, SHA-1 piece hashes). Tạo file `.torrent` tại runtime tốn CPU và I/O không cần thiết.

§3b giải quyết: **generate `.torrent` files ngay tại build time**, lưu cạnh `.hash` files. Runtime chỉ cần đọc và seed.

Khi một directory chỉ có subdirs (không có JSON files trực tiếp), code cũ vẫn tạo `_.torrent`. Nhưng tạo từ cái gì? Từ hash string của child hashes — một chuỗi text thuần túy, không phải content thật.

```js
if (fileDescriptors.length > 0) {

}
```

### ⚔️ Boss 8 — Kẻ Phá Hoại Binary: `FS.write(Uint8Array)` → JSON

**Cấp độ:** 🔴 Critical

`wrapTorrent()` tạo ra một `Uint8Array` — binary bencode data. `FS.write()` được gọi để ghi xuống disk.

Nhưng `write.js` không biết xử lý `Uint8Array`. Nó nhận object, không nhận ra là binary, và làm điều tệ nhất có thể:

```js
data = JSON.stringify(content)
// → '{"0":100,"1":52,"2":58,...}'
```

Mọi file `.torrent` được ghi ra disk đều là JSON của byte values. Mở file ra đọc: `{"0":100,"1":52}`. WebTorrent không thể parse. Toàn bộ §3b vô nghĩa.

**Hạ gục:** Binary shortcut trước mọi serialization logic:

```js
if (content instanceof Uint8Array) {
    if (BROWSER) {
        await opfs.write(path, content)
        return { success: true, path: path.join("/") }
    }
    fs.writeFileSync(filePath, content)  // không có encoding = binary mode
    return { success: true, path: filePath }
}
```

`Uint8Array` đến, `Uint8Array` được ghi. Không qua JSON, không qua bất cứ thứ gì khác.

## Chương V — Kẻ Giữ Cổng: Trackers & Torrent Thread

### Bối cảnh

P2P cần trackers — những server trung gian giúp peers tìm thấy nhau. Browser chỉ support WebSocket trackers (`wss://`). Toàn bộ danh sách tracker công cộng lớn nhất thế giới (ngosang/trackerslist) hiện chỉ còn **1 tracker `wss://`** còn active.

§4 xây hệ thống tracker management: fetch từ ngosang, merge với fallback cứng, lưu vào YAML. §5 đưa toàn bộ seeding logic vào worker thread riêng biệt — không block UI, không block business logic.

---

### ⚔️ Boss 9 — Kẻ Xóa Ký Ức: Fallback Trackers Bị Mất

**Cấp độ:** 🟡 Medium

`builder/trackers.js` chạy, fetch từ ngosang, và ghi kết quả vào `trackers.yaml`. Logic đơn giản. Nhưng có một tình huống chưa được xử lý:

```js
const trackers = [...new Set([...fetched])]  // ← chỉ fetched, không có fallback
await FS.write([...paths.src.statics, "trackers.yaml"], trackers)
```

Nếu network fail, `fetched = []`. Script ghi `[]` vào file. **Tất cả 4 trackers trong `trackers.yaml` bị xóa.** Lần chạy tiếp theo: `Construct.Torrent()` nhận mảng rỗng, không announce đến tracker nào, không tìm được peer nào.

*Một lần network hiccup trong CI — toàn bộ P2P layer bị tắt cho đến khi ai đó nhận ra và chạy lại script.*

**Hạ gục:** Hardcode fallbacks vào script — luôn merge, không bao giờ lose:

```js
const FALLBACK = [
    "wss://tracker.btorrent.xyz",
    "wss://tracker.openwebtorrent.com",
    "wss://tracker.webtorrent.dev",
]
const trackers = [...new Set([...fetched, ...FALLBACK])]
```

Fallbacks là safety net. Dù ngosang sập, dù network fail, ba cái này luôn có mặt.

### ⚔️ Boss 10 & 11 — Hai Boss Cuối: Đang Chờ Policy

Hai boss này không thể hạ gục bằng code. Chúng cần **quyết định chiến lược**.

#### Boss 10 — Bóng Ma GDPR: IP Leak Không Có Consent

Khi `Construct.Torrent()` chạy — ngay khi user mở app — browser kết nối đến 4 public trackers. Tracker nhận IP của user. IP = personal data dưới **GDPR Article 4(1)**.

Không có consent banner. Không có opt-in checkbox. Không có privacy policy update.

*Một user ở EU mở akao lần đầu: IP họ đã được broadcast đến tracker ở US, không ai hỏi, không ai thông báo.*

## Lời Kết

Chín boss đã ngã. Hai boss cuối đang đứng chờ — không phải vì thiếu vũ khí, mà vì trận chiến đó cần **quyết định của con người**, không phải code.

Codebase này giờ đây có một nền tảng vững chắc: FS đọc được mọi loại file, torrent state không bao giờ bị leak, hash phản ánh đúng thực tế, binary data không còn bị corrupt ngầm.

Hành trình P2P còn dài. Nhưng những bước đầu tiên — được đặt xuống đúng chỗ.
