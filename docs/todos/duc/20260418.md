# Tool Kit QA Usage — `tool-kit-test` and `tool-kit-attack`

> **Direction (Duc, 2026-04-26):** QA surface is exactly two self-contained tool-kit skills:
>
> 1. **`tool-kit-test`** — audits one feature through five coverage flows, one flow at a time, with a user gate after each.
> 2. **`tool-kit-attack`** — attacks one feature, writes attack specs, runs them, then asks for approval before touching source.
>
> No old entrypoint names in active usage docs. No pipeline runner. No preview/review/QA meta-skill. Each skill must be callable on its own and must not depend on a Node orchestrator.

## 2. Usage Guide

Both skills are feature-scoped. Pass exactly one feature name unless a skill explicitly documents an option. A feature can resolve to either:

- a route under `src/UI/routes/<feature>/`
- a core module under `src/core/<feature>/`, `src/core/<feature>.js`, or `src/core/<sub>/<feature>.js`

The source of truth for installable skills is `tools/tool-kit/skills/`. Persona checklists live under `tools/tool-kit/personas/` and are reference material for `tool-kit-attack`, not standalone skills.

### `tool-kit-test` — sequential feature coverage

Use:

```bash
$tool-kit-test <feature>
$tool-kit-test <feature> --solo
$tool-kit-test <feature> --start <N>
```

`tool-kit-test` walks one feature through five flows. It audits all flows up front, consults coverage/spec/TDD lenses unless `--solo` is passed, then executes one flow at a time. After each flow it stops and waits for the user to choose `continue`, `end`, `skip to <M>`, `redo`, or `force`.

This gate is load-bearing. The skill must not silently skip from flow 1 to flow 5 just because it thinks a flow is not applicable. `not-applicable` is a hypothesis the user can override with `force`.

| # | Flow | Where it lives | What "this feature has it" means |
|---|---|---|---|
| 1 | Build artifacts | Inline skill assertion | Route features must have `build/UI/routes/<feature>/index.html` and `index.js` with the expected custom element. Core modules must have `build/core/<path>/<feature>.js`. |
| 2 | Isomorphic core / logic | `src/core/tests/<feature>.test.js` | A focused non-DOM test registered in `src/core/tests/manifest.js`, run as a single module. |
| 3 | Web components / Shadow DOM | `e2e/components/<feature>.spec.ts` | Render, reactivity, custom events, lifecycle. Use Shadow DOM pierce paths from the actual route/component shape. |
| 4 | Local-first / cache | `e2e/local-first/<feature>.spec.ts` | Offline round-trip via `page.context().setOffline(true)` plus cache/hash invalidation where applicable. |
| 5 | User journeys | `e2e/<feature>.spec.ts` | End-to-end user behavior against `build/`, served through `preview.js`. |

Run commands are always scoped:

- Flow 1: inline `existsSync` / `readFileSync` assertion in the skill.
- Flow 2: `node --input-type=module -e "import('./src/core/tests/<feature>.test.js').then(m => m.default ? m.default() : null)"`
- Flow 3: `npx playwright test --project=chromium --grep "@<feature>" e2e/components/`
- Flow 4: `npx playwright test --project=chromium --grep "@<feature>" e2e/local-first/`
- Flow 5: `npx playwright test --project=chromium --grep "@<feature>" e2e/<feature>.spec.ts`

Zero-match is not a pass. If a spec exists but has no `@<feature>` tag, status is `no-specs`. `scaffolded` and `no-specs` count as TODO in the verified ratio.

Final report shape:

```text
feature=<name>  flows-walked=<list>  flows-skipped=<list>
  flow1=pass
  flow2=pass
  flow3=n/a
  flow4=pass
  flow5=scaffolded
verified = passed / (passed + failed + scaffolded + no-specs)

Suggested next:
  $tool-kit-test <sibling> — reason
```

### `tool-kit-attack` — feature-scoped vulnerability hunt

Use:

```bash
$tool-kit-attack <feature>
$tool-kit-attack <feature> --solo
```

`tool-kit-attack` resolves one feature, reads source/docs, picks applicable personas, writes or refreshes attack specs in `e2e/attacks/`, runs only this feature's attack specs, then emits an approval-gated attack plan. Passing attack tests mean the exploit succeeded.

Source files are frozen until the user explicitly replies `apply all` or `apply <N>`. The skill may create or update attack specs before approval, but it must not modify `src/`, `playwright.config.ts`, `package.json`, or `.github/workflows/` before the decision gate.

Persona selection:

| Signal in feature source | Class | Vectors |
|---|---|---|
| Always | Alex + Rex baseline | OWASP/CWE checks, input abuse, state races, auth bypass |
| `estimateGas`, `Wallets[`, `slippage`, `SendLogic`, `swap`, `withdraw`, `deposit` | Vera / Web3 | slippage, deadline, stale quote, approval, chain switch, encoding |
| Auth, session, token, password | Auth | session fixation, token replay, weak guard, brute-force surface |
| Form submit + async | Race | TOCTOU, double-submit, rapid-fire replay |
| User input to DOM | Injection | XSS reflected/stored/DOM, unicode confusables, prototype pollution |
| Money flow + user input | Cora / confused user | paste malformed address, double-tap submit, locale-formatted numbers, back-button mid-tx |
| Heavy async work, large dependencies, expensive loops | Pax / performance | resource exhaustion, unbounded work, route load budget issues |

Attack specs must use this header shape:

```ts
// Vector: <vector-name>
//
// Feature: <feature>
// Target:  /en/<feature> -> <component-or-surface>
//
// Threat model:
//   Actor:      <who>
//   Capability: <what they can do>
//   Goal:       <what they're trying to break>
//
// CWE:    CWE-<NN>
// MITRE:  T<NNNN>
// Severity: <critical | high | medium | low>
//
// Status semantics:
//   Test passes -> exploit succeeded -> vulnerability confirmed -> fix needed
//   Test fails  -> defense held
```

Run command:

```bash
npx playwright test --project=chromium --grep "@<feature>" e2e/attacks/
```

Plan report shape:

```markdown
## Attack Plan: <feature>

**Surface scanned:** <classes>
**Vectors run:** <N>   **Confirmed vulnerabilities:** <K>   **Defenses held:** <N - K>

---

### Finding 1 — <vector-name> (severity: HIGH)

- **CWE:** CWE-20
- **MITRE:** T1499
- **Spec:** `e2e/attacks/<feature>-<vector>.spec.ts:NN` (passed -> exploit succeeded)
- **Source:** `src/path/to/file.js:LINE`
- **Threat:** <one-sentence threat model>

**Proposed fix:**

```diff
- get slippage() { return Number(this.$slippage?.value) || 0.5 }
+ get slippage() {
+     const raw = Number(this.$slippage?.value)
+     if (!Number.isFinite(raw) || raw <= 0) return 0.5
+     return Math.min(Math.max(raw, 0.1), 50)
+ }
```

**Why this fix:** <one short paragraph explaining what the fix prevents and what test flips after the fix>.

---

## Awaiting decision

Reply one of:
- `apply all` — I write all proposed fixes.
- `apply 1, 3` — I write only the listed findings.
- `skip` — I write nothing; you'll handle it manually.
- `explain N` — I expand on finding N before you decide.

I will not modify source files until you reply.
```

After approval:

- Freeze each approved attack spec by recording its bytes / sha256.
- Apply only the approved source diff.
- Re-check that the spec stayed byte-identical.
- Re-run the approved attack spec with `npx playwright test --project=chromium e2e/attacks/<feature>-<vector>.spec.ts`.
- The attack spec must flip from pass to fail. If it does not flip, revert the source diff and report the unexpected result.

Hard constraints for both skills:

- Act on one feature only.
- Never call `npm test`, `npm run test:isomorphic`, `npm run test:browser`, `npm run test:runtime`, `npm run test:playwright`, or unfiltered `npx playwright test`.
- All Playwright specs target built artifacts through `preview.js`.
- Never commit `test.only`.
- Keep reports grounded in this codebase; no generic security checklist output.

## 3. Install / Verify

Install:

```bash
node tools/tool-kit/install.js --targets all
```

Uninstall:

```bash
node tools/tool-kit/uninstall.js
```

The installer must remain idempotent and non-interactive when `--targets` is passed. The uninstaller may clean old entrypoints from earlier installs, but that cleanup is implementation detail, not active usage guidance.

Expected install targets:

| Agent | Install path | Layout |
|---|---|---|
| Claude Code | `~/.claude/skills/<skill>/SKILL.md` | per-skill dir |
| Codex CLI | `~/.codex/skills/<skill>/SKILL.md` | per-skill dir |
| Cursor | `~/.cursor/skills/tool-kit/<skill>.md` | flat bundle |
| Windsurf | `./.windsurf/skills/tool-kit/<skill>.md` | flat bundle |
| GitHub Copilot | `./.github/skills/tool-kit/<skill>.md` | flat bundle |

Verify:

1. `node tools/tool-kit/uninstall.js`
2. `node tools/tool-kit/install.js --targets all`
3. Confirm only `tool-kit-test` and `tool-kit-attack` are installed as active skills.
4. In a fresh Codex session, invoke `$tool-kit-test swap`.
5. Confirm it audits `swap`, executes flow 1, then stops at the decision gate.
6. Invoke `$tool-kit-attack swap`.
7. Confirm it writes or refreshes `e2e/attacks/swap-*.spec.ts`, runs `npx playwright test --project=chromium --grep "@swap" e2e/attacks/`, and emits the approval prompt before any source edit.

## 5. Follow-up — Tag Discipline Story

**Story:** As a developer running `$tool-kit-test <feature>`, I need `@<feature>` tag presence enforced at invocation time so a missing or misspelled tag fails fast with a clear error rather than silently running zero tests.

**Acceptance criteria:**

1. Skill checks that `npx playwright test --grep "@<feature>" --list` returns at least one named test before invoking the actual run.
2. Zero-match emits a named error referencing the spec file and the missing tag.
3. Tag-naming convention added to `tools/tool-kit/README.md`: lowercase feature name, single token, `@flowN @<feature>` pair on the outermost `describe`.
4. Optional lint rule or pre-commit hook fails when a new `e2e/**/*.spec.ts` is added without a `@flowN @<feature>` describe tag.

**Why:** `@<feature>` tag discipline is now load-bearing. If devs forget the tag on new specs, grep selection silently misses them. The skill contract says `no-specs` counts as TODO, not pass; this story turns that contract into a hard gate.

## 6. Open Questions

1. Should `tool-kit-attack` always wait for explicit approval, or later accept a guarded batch mode? Recommendation: always gate by default.
2. Should `scan-components.js` stay as a developer utility only, or become a required helper for flow 3 scaffolding? Recommendation: keep optional.
3. For flow 1, per-feature checks should assert only files the feature owns. Global build outputs are out of scope for a single-feature test run.
