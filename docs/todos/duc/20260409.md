# DB.js — Data Sync to SQL/OPFS

**Date:** 2026-04-09
**Author:** Duc

---

## Mục Tiêu Phase Này

Sync toàn bộ item catalog vào **SQLite WASM trên OPFS** để query được offline, cross-locale, cross-game.

- Server → IDB (cache) → SQL/OPFS (query layer)
- Chạy hoàn toàn background, không block UI
- Hash validation đảm bảo data fresh

---

## Kiến Trúc 3 Lớp

```
SERVER (static files)
    │  fetch + validate hash
    ▼
IDB (IndexedDB)          ← cache JSON, source of truth cho UI
    │  fire-and-forget
    ▼
SQLite WASM (OPFS)       ← /app.db, file thực sự trên filesystem
```

**IDB** có thể bị browser clear (privacy mode, quota). **OPFS không bị** — app vẫn query được dù IDB trống.

**Correctness** do hash validation đảm bảo, không phải OPFS:
- Hash match → serve từ IDB (đã validate lần trước)
- Hash miss → fetch server → update IDB → sync SQL

---

## Flow Tổng Quát

```
App gọi DB.get(path)
    │
    ├─ DEV mode: fetch server → write IDB → $syncToSQL → return
    │
    └─ PROD mode:
          load hash từ IDB (cached)
          fetch .hash từ server (lightweight)
          │
          ├─ MATCH  → load từ IDB → $syncToSQL → return
          └─ MISS   → fetch server → write IDB → $syncToSQL → return

SQL ready (background):
    _yield() → $rebuildFromIDB() → _yield() → $syncCatalog()
```

## Các Hàm Chính

### `DB.get(path)` — Entry point

Tất cả data đi qua đây. Trả về data ngay, SQL sync chạy ngầm (fire-and-forget).

### `$syncToSQL(path, data)` — Ghi vào SQL

Không ghi trực tiếp — push vào `_pending[]`, `queueMicrotask` gom tất cả ops trong cùng 1 tick thành **1 batch transaction → 1 fsync**.

```
200 items fetch song song trong 1 tick
    → 200 $syncToSQL() calls
    → 1 queueMicrotask
    → 1 db.batch() → 1 fsync
```

10-50× nhanh hơn so với 200 `db.run()` riêng lẻ.

Dùng `INSERT OR REPLACE` — luôn ghi data mới nhất.

### `_syncInsert(path, data)` — Dùng trong rebuild

Giống `$syncToSQL` nhưng dùng `INSERT OR IGNORE` — skip nếu row đã có. Không overwrite data đúng.

### `$rebuildFromIDB()` — Chạy khi SQL start

Đọc tất cả keys từ IDB, insert vào SQL những gì bị thiếu.

```
Đọc tất cả IDB keys
    │
    Count guard (O(1)):
      game_items COUNT ≥ gameKeys trong IDB?
      catalog_items COUNT ≥ catalogKeys trong IDB?
      │
      ├─ ĐỦ RỒI → _yield() → $syncCatalog()
      │
      └─ THIẾU → xử lý từng chunk 30 keys (idle time)
                 INSERT OR IGNORE từng item
                 → _yield() → $syncCatalog()
```

Chunk 30 + `_yield()` giữa mỗi chunk → browser không bị block.

### `$syncCatalog()` — Full catalog sync

Chạy sau `$rebuildFromIDB`. Đảm bảo **tất cả** items × **tất cả** locales có trong SQL.

Cần thiết vì: SQL primary key là `(id, game_id, locale)`. Nếu user chỉ dùng `vi`, SQL chỉ có `vi` rows. Search `WHERE locale='zh'` → rỗng. `$syncCatalog` download đủ 18 locales × 1534 items.

```
Load locales.json → 18 locales
Load items/meta.json → { pages: 154, children: 1534 }

Count guard: sqlTotal >= 1534 × 18 → skip (đã đủ)

Chưa đủ → walk từng page (154 pages):
    _yield()
    DB.get("items/{page}.json") → list item keys
    for each item × each locale:
        _yield()
        DB.get("items/{gameId}/{itemId}/{locale}.json")
        → hash check → IDB → $syncToSQL → SQL
```

### `_flush()` — Batch write

```
Lấy tất cả ops từ _pending[]
Ensure schemas (CREATE TABLE IF NOT EXISTS)
    │
    ├─ Fast path: db.batch(all ops) → 1 fsync
    │
    └─ Nếu batch fail (1 op bad data):
           fallback: run từng op riêng
           op lỗi → warn + skip
           ops đúng → vẫn được lưu
```

### `_yield()` — Nhường browser

Không dùng `requestIdleCallback` (unreliable, Safari bug). Dùng fallback chain:

```
scheduler.yield()  →  MessageChannel  →  setTimeout(0)
```

## SQL Worker (FIFO Queue)

Mọi SQL call đi qua `src/core/SQL/call.js` — FIFO serialize:
- 1 command in-flight tại một thời điểm
- Watchdog 10s → nếu Worker không trả lời → reject, tiếp tục queue

Không bao giờ flood SQL Worker.
