# SQL Integration — 2026-03-28 (Duc)

## Bối cảnh

Dự án có một lớp `DB.js` bọc IndexedDB (IDB) — dùng để cache data tĩnh từ server (items, routes...).
IDB chỉ là key-value store nên muốn filter phải load toàn bộ về JS rồi mới lọc.

**Vấn đề cụ thể:** route-game gọi `_loadAllRemaining()` để kéo hết 500+ items về trước khi filter theo type/rarity/tên
→ chậm, tốn memory, không scale được.

**Giải pháp:** mỗi khi `DB.get()` fetch data mới từ server (phát hiện qua hash mismatch),
mirror data đó vào SQLite song song IDB. Sau đó query bằng SQL thay vì load hết.

---

## Kiến trúc sau khi implement

```
Server (JSON files)
       │  hash mismatch → fetch mới
       ▼
   DB.get()  ──────────────────────────────────────────────┐
       │ await                                              │ fire-and-forget (không await)
       ▼                                                    ▼
   IDB (IndexedDB)                               DB.$syncToSQL(path, data)
   [source of truth]                                        │
                                                   transformer(path, data)
                                                            │
                                                   INSERT OR REPLACE
                                                            ▼
                                                   SQLite (OPFS) [query cache]
                                                            │
                                                   DB.query("SELECT ...")
                                                            ▼
                                                      Component UI
```

**Nguyên tắc quan trọng:**
- IDB là source of truth — SQLite chỉ là cache để query nhanh
- SQL sync không bao giờ block `DB.get()` — UI không bị chậm dù SQL có lỗi
- SQL fail → chỉ `console.warn`, component fallback về `_loadAllRemaining()` từ IDB

## Cách hoạt động — ví dụ xuyên suốt

Lấy ví dụ: user mở trang game Arc Raiders, app fetch item "Kiếm Lửa".

**Path trong IDB:**
```
["statics", "items", "arc-raiders", "sword-01", "vi.json"]
```

**Data JSON từ server:**
```json
{ "name": "Kiếm Lửa", "rarity": "rare", "type": "weapon", "value": 1500, "images": ["thumb.webp"] }
```

### Bước 1 — DB.get() phát hiện hash mismatch, fetch data mới

```js
// DB.js
const data = await load(path)
await Indexes.Statics.get(path).put(data)  // lưu IDB
DB.$syncToSQL(path, data)                   // không await — chạy ngầm
return data                                 // trả về ngay cho UI
```

### Bước 2 — transformer nhận path + data, trả về SQL operation

```js
// transformer.js
transform(
    ["statics", "items", "arc-raiders", "sword-01", "vi.json"],
    { name: "Kiếm Lửa", rarity: "rare", ... }
)

// → trả về:
{
    schema: SCHEMA_GAME_ITEMS,   // DDL CREATE TABLE (chỉ gửi lần đầu/session)
    upsert:  "INSERT OR REPLACE INTO game_items (id,game_id,locale,...) VALUES (?,?,?,...)",
    values:  ["sword-01", "arc-raiders", "vi", "Kiếm Lửa", "rare", "weapon", 1500, ...]
}
```

Transformer phân biệt loại item qua **độ dài của `rest`** (phần path sau `"statics","items"`):

| Path | rest.length | Bảng |
|------|------------|------|
| `["statics","items","arc-raiders","sword-01","vi.json"]` | 3 | `game_items` |
| `["statics","items","item-99","vi.json"]` | 2 | `shop_items` |

### Bước 3 — DB.$syncToSQL chạy ngầm, INSERT vào SQLite

```js
DB.sql()           // lấy SQL singleton (khởi tạo lần đầu nếu chưa có)
  .then(db => {
      // lần đầu: gửi DDL lên worker, ghi vào _schemas để lần sau skip
      // lần sau: nhảy thẳng vào INSERT
      return db.run(op.upsert, op.values)
  })
  .catch(err => console.warn("[DB.$syncToSQL]", path, err.message))
  // SQL fail → chỉ warn, không ảnh hưởng gì đến UI
```

### Bước 4 — Component query SQL thay vì load hết

```js
// thay vì _loadAllRemaining() + filter JS:
const rows = await DB.query(
    "SELECT * FROM game_items WHERE game_id=? AND locale=? AND rarity=?",
    ["arc-raiders", "vi", "rare"]
)
```

## Các vấn đề phát sinh trong quá trình implement

### V1 — Static import SQL.js break `npm test`

SQL.js dùng `/core/Threads.js` (absolute path, chỉ resolve trong browser).
Static import kéo nó vào chain: `Forex.js → DB.js → SQL.js → Threads.js` → Node không tìm thấy → test crash.

**Fix:** dynamic import bên trong `sql()` — chỉ load khi thực sự gọi trong browser:
```js
const { default: SQL } = await import("./SQL.js")
```

---

### V2 — `$syncToSQL` với `await` làm chậm `DB.get()`

```js
// Sai — UI phải đợi SQL init xong (100-500ms lần đầu) trước khi nhận data
await DB.$syncToSQL(path, data)
return data

// Đúng — UI nhận data ngay, SQL chạy ngầm
DB.$syncToSQL(path, data)   // không await
return data
```

---

### V3 — Naming convention sai

Project dùng `$` prefix cho internal dispatch methods (`$call`, `$emit`, `$root`...).
`_syncToSQL` → đổi thành `$syncToSQL`.

---

### V4 — `data.id` null → INSERT fail im lặng

Shop item path: `["statics","items","item-99","vi.json"]` — `itemId = "item-99"` lấy từ path.

Nếu JSON không có field `id`:
```js
// Sai
values: [data.id, ...]        // data.id = undefined → PK = null → INSERT fail, .catch() nuốt lỗi

// Đúng
values: [data.id ?? itemId, ...] // fallback về "item-99" từ path
```

---

### V5 — Schema DDL gửi lên worker mỗi lần sync

500 items sync = 500 lần gửi `CREATE TABLE IF NOT EXISTS` lên SQL worker — thừa hoàn toàn.

**Fix:** `static _schemas = new Set()` — track schema nào đã init trong session:
```js
const ready = DB._schemas.has(op.schema)
    ? Promise.resolve()                                          // đã init → skip
    : db.exec(op.schema).then(() => DB._schemas.add(op.schema)) // chưa init → tạo rồi ghi nhớ
```

---

### V6 — Race condition: `sql()` async → nhiều SQL instance

```js
// Vấn đề: sql() là async function
static async sql() {
    if (!DB._db) {           // ← concurrent calls đều pass qua đây
        DB._db = new SQL()   // ← nhiều instance được tạo
        await DB._db.ready
    }
}

// Fix: sql() là sync, gán Promise TRƯỚC khi await bất kỳ thứ gì
static sql() {
    if (!DB._db) {
        DB._db = (async () => { ... })()  // ← gán ngay, concurrent calls đều nhận cùng Promise
    }
    return DB._db
}
```

---

### V7 — Rejected Promise lưu mãi trong `_db`

Nếu SQL worker fail (WASM không load, OPFS lỗi, timeout 30s):
```js
DB._db = rejectedPromise   // truthy → if (!DB._db) không bao giờ reset
                           // → mọi DB.query() sau đó fail mãi đến khi refresh trang
```

**Fix:**
```js
DB._db = (async () => { ... })()
    .catch(err => {
        DB._db = null   // reset → lần sau retry được
        throw err
    })
```

**Về timeout:** `SQL/call.js` có timeout cứng 30s cho mọi worker call.
Sau 30s → reject → `.catch()` bắt → `_db = null` → retry được. Không bao giờ hang mãi.

## Nợ kỹ thuật còn lại

### Nợ 1 — Tạo `DB/queries/` — pattern query tập trung (ưu tiên cao)

Hiện tại component phải tự viết SQL string. Cần tạo lớp query functions:

```
src/core/DB/queries/
├── game-items.js    → export queryGameItems(gameId, locale, { type, rarity, search, sort })
└── shop-items.js    → export queryShopItems(locale, { category, search, maxPrice })
```

Ví dụ dùng:
```js
// Thay vì component tự viết SQL:
const rows = await DB.query("SELECT * FROM game_items WHERE game_id=? AND locale=? AND rarity=?", [...])

// Component chỉ gọi named function, không biết SQL tồn tại:
const items = await queryGameItems("arc-raiders", "vi", { rarity: "rare" })
```

---

### Nợ 2 — Thay filter trong route-game bằng SQL (ưu tiên cao)

File: `src/UI/routes/game/[game]/index.js`

```js
// Hiện tại (chậm):
await _loadAllRemaining()   // kéo hết 500+ items
items = items.filter(i => i.rarity === filter.rarity && i.type === filter.type)

// Sau khi fix (nhanh):
const items = await queryGameItems(gameId, locale, filters)
if (!items.length) {
    // SQL chưa populate (app mới mở, hoặc vừa migrate) → fallback IDB
    await _loadAllRemaining()
    items = applyFilters(items, filters)
}
```

---

### Nợ 3 — Fallback khi SQL rỗng sau khi app khởi động (ưu tiên trung bình)

Hiện tại mỗi lần app khởi động, SQL DROP toàn bộ bảng và recreate từ đầu.
SQLite rỗng cho đến khi `update.js` repopulate từ IDB (có thể mất vài giây).

Trong khoảng thời gian đó nếu component query SQL → nhận về mảng rỗng → phải fallback:

```js
const rows = await queryGameItems(...)
if (!rows.length) {
    // SQL chưa có data — fallback về IDB
    return _loadAllRemaining().then(applyFilters)
}
```

Đây là **known behavior, không phải bug** — chỉ cần handle đúng ở caller.

---

### Nợ 4 — Schema versioning: hiện tại DROP toàn bộ mỗi lần init (ưu tiên thấp)

Hiện tại không có version tracking — mỗi lần khởi động đều DROP hết:
```js
await db.exec(ALL_TABLES.map(t => `DROP TABLE IF EXISTS ${t};`).join("\n"))
```

**Hệ quả:** SQLite không giữ lại cache giữa các session → luôn phải repopulate từ IDB.

**Cần làm sau:** dùng `PRAGMA user_version` hoặc bảng `_meta(table, version)` để chỉ DROP khi schema thực sự thay đổi. Khi thêm bảng mới, chỉ cần thêm tên vào `ALL_TABLES` trong `transformer.js` — `DB.js` tự loop DROP, không cần sửa tay.

**Lưu ý:** `PRAGMA user_version` là per-database (toàn bộ file `shop.db` dùng chung 1 số) — khi bump version thì tất cả bảng đều bị DROP cùng lúc, không selective được từng bảng.
