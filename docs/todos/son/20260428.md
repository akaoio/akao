# 20260428

## AKAO production + game item IDs + geo cache

Hôm nay tập trung vào ba mảng lớn:

1. dựng production flow riêng cho `akao`
2. sửa triệt để cách đặt ID cho game items
3. chặn các đợt geo rebuild vô ích

## 1. Production mode cho akao đã được dựng riêng

`akao` trước đó chưa có production entrypoint thật sự.

Trong khi đó `zen` đang chạy production và có logic dùng cert ở `~/cert.pem` + `~/key.pem`, dễ đụng nhau nếu `akao` cũng đi theo lối đó.

Đã thêm:

- `prod.js`
- `start:prod`
- `prod`

Production server mới của `akao`:

- serve static `build/`
- bind `443`
- redirect `80 -> 443`
- **không** fallback sang cert ở home directory
- mặc định dùng cert riêng ở:
  - `.certs/prod-key.pem`
  - `.certs/prod-cert.pem`

Điểm quan trọng nhất:

> production của `akao` đã được tách hẳn khỏi cert strategy của `zen`.

## 2. SSL setup cho akao đã được dựng thành flow riêng

Đã thêm các script:

- `script/ssl.sh`
- `script/install-service.sh`
- `script/setup-prod.sh`
- `script/challenge-server.js`

Mục tiêu là để `akao` có thể:

- tự lấy Let's Encrypt cert riêng
- không đụng cert của `zen`
- cài service riêng bằng systemd

Ban đầu flow dùng `acme.sh --standalone` bị fail vì validation qua HTTP challenge không thực sự có server giữ port `80` ổn định theo cách cần thiết.

Sau đó flow đã được đổi thành:

1. cài service nhưng chưa start
2. mở temporary challenge server trên `80`
3. issue cert bằng **webroot mode**
4. xong mới start `akao`

## 3. Đã sửa bug ACME challenge qua IPv6

Có một lỗi khá khó chịu:

- Let's Encrypt verify qua IPv6
- challenge path trả `500`

Nguyên nhân là `challenge-server.js` parse URL dựa trên `HOST="::"`, làm request path hỏng trong vài trường hợp IPv6.

Đã sửa để server đọc trực tiếp path từ request thay vì dựng base URL từ `HOST`.

Sau khi sửa:

- test qua IPv4: OK
- test qua IPv6: OK

## 4. Đã tìm ra vì sao akao.io vẫn không lên sau khi có cert

HTTPS public không lên không phải vì cert.

Root cause thật sự là:

```text
Error: listen EACCES: permission denied :::443
```

Tức là service `akao` chạy dưới user thường nhưng chưa có quyền bind privileged ports.

Đã vá generator của systemd unit để thêm:

- `AmbientCapabilities=CAP_NET_BIND_SERVICE`
- `CapabilityBoundingSet=CAP_NET_BIND_SERVICE`

Kết luận:

> blocker chính của `80/443` là quyền bind port, không phải firewall ở giai đoạn đó.

## 5. Geo build đã có fingerprint cache

Trước đó phần GeoNames dù dữ liệu đầu vào không đổi vẫn có xu hướng chạy lại một đống bước nặng:

- parse countryInfo
- parse feature codes
- quét allCountries
- generate lại hash cây thư mục

Đã thêm fingerprint cache trong `builder/geo/build.js`.

Fingerprint hiện tại dựa trên:

- `countryInfo.txt`
- `featureCodes_en.txt`
- `allCountries.txt`
- `hierarchy.txt`
- `adminCode5.txt`
- mode build/test
- output base

Nếu fingerprint không đổi và output cần thiết đã tồn tại, build sẽ thoát sớm.

Đã xác nhận lần chạy sau sẽ báo:

```text
⊘ Geo inputs unchanged; skipping rebuild.
```

## 6. Cách đặt ID cho game items đã được làm lại lần thứ nhất

Trước đó item IDs của game bị bẩn vì suffix phụ thuộc quá nhiều vào raw payload.

Điều này gây ra:

- cùng một item nhưng ra nhiều folder khác nhau
- duplicate raw ids
- build churning vô ích

Đã có một pha sửa trung gian để:

- chặn duplicate raw ids
- chặn raw.yaml rewrite do `null` / missing / key order

Nhưng sau đó yêu cầu được đổi sang một rule sạch hơn nữa.

## 7. Rule ID mới cho game items đã được chốt

Sau khi trao đổi lại, quyết định cuối cùng là:

- xóa sạch game items trong `src/statics/items/{arc-raiders,diablo-4}`
- bỏ ý tưởng preserve folder cũ
- rebuild lại từ đầu theo rule mới

Rule mới:

- nếu slug **không trùng** trong cùng game -> dùng luôn slug, **không suffix**
- nếu slug **trùng** -> sort theo hash từ identity object đã lọc field ổn định, rồi đánh:
  - `name-0`
  - `name-1`
  - `name-2`

Identity object để sort duplicate chỉ giữ những field ổn định đủ để phân biệt item:

- `gameId`
- `sourceId`
- `name`
- `type`
- `subtype`
- `rarity`
- `slots`
- `classes`
- `icon`

Điểm quan trọng:

> suffix hash dài đã bị bỏ khỏi folder name; hash chỉ còn đóng vai trò làm khóa sort ổn định trong nhóm duplicate.

## 8. Game items đã được rebuild sạch từ đầu

Đã xóa sạch:

- `src/statics/items/arc-raiders`
- `src/statics/items/diablo-4`
- `build/statics/items/arc-raiders`
- `build/statics/items/diablo-4`

Sau đó build lại:

- `node build.js games --prune`
- `node build.js core`
- `node build.js index`

Kết quả:

- `arc-raiders`: đa số item là slug sạch, không suffix
- `diablo-4`: chỉ các nhóm duplicate mới có suffix số

Ví dụ:

- `adrenaline-shot`
- `acoustic-guitar`
- `anvil-i`
- `blue-rose-0`
- `blue-rose-1`
- `blue-rose-2`
- `blue-rose-3`

## 9. Tính deterministic của game item tree đã được xác nhận

Sau khi đổi ID rule mới, đã đo lại checksum trước/sau khi rerun `node build.js games --prune`.

Kết quả:

- `src: stable`
- `build: stable`

Điều này có nghĩa là:

> với cùng input hiện tại, build lại sẽ không làm đổi item trees nữa.

## 10. Kết luận hôm nay

Nếu phải tóm gọn ngày hôm nay:

> 20260428 là ngày production path của `akao` được tách riêng khỏi `zen`, ACME flow được dựng lại cho đúng mô hình HTTP challenge, geo build được gắn fingerprint cache để thôi rebuild vô ích, và toàn bộ game item IDs được thiết kế lại sang cấu trúc sạch hơn: item thường thì không suffix, item trùng slug mới dùng suffix số nguyên ổn định.

Việc còn lại không còn là “ID có bị rung nữa không”.

Phần đó đã được chốt.

Việc còn lại chủ yếu là:

- restart service `akao` với systemd unit mới có bind capability
- verify public `80/443`
- tiếp tục refine game data nếu sau này cần thêm per-item custom config thật trong `src`
